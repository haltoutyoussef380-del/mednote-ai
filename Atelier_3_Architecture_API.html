Atelier - Architecture & API du MVP MedNote AI

Partie 1 : Identifier les APIs du MVP

Voici les actions principales par profil, chaque action correspondant à un endpoint API.

Pour un Utilisateur (Staff Médical : Médecin, Infirmier, Secrétaire)

  • Se connecter (Login)
  • Consulter la liste des patients
  • Créer une fiche patient (Secrétaire/Médecin)
  • Créer une observation médicale (Médecin)
  • Dicter une note via IA (Médecin)
  • Ajouter des constantes/soins (Infirmier)
  • Rechercher un patient

Pour un Administrateur (Super Admin)

  • Créer un compte staff (Onboarding Médecin/Infirmier/Secrétaire)
  • Lister le personnel
  • Supprimer une note (Cas exceptionnel/Erreur)
Note importante : Il n'y a PAS d'inscription publique pour les profils médicaux. Les comptes sont créés exclusivement par le Super Admin.

Partie 2 : Définir les Endpoints

Note: Authentification via Header Authorization: Bearer <TOKEN> pour tous les endpoints sauf /auth/login

Endpoint (URL) Méthode Rôle Autorisé Données Envoyées Données Reçues
/auth/login POST Public { email, password } { access_token, user_role }
/api/admin/users POST 👑 Super Admin { email, password, role, name } { user_id, created_at }
/api/patients GET Tous (sauf public) { page: 1, search: "nom" } [ { id, nom, prenom... } ]
/api/patients POST 🩺 Médecin, 📅 Secrétaire { nom, prenom, secu, contact } { patient_id }
/api/patients/{id} PUT 🩺 Médecin, 📅 Secrétaire { adresse, tel } { success: true }
/api/notes POST 🩺 Médecin { patient_id, type, content } { note_id, created_at }
/api/care POST 🩺 Médecin, 💉 Infirmier { patient_id, type, value, unit } { care_id }
/api/voice-pilot POST 🩺 Médecin { audio_blob, context_step } { transcript, actions: [] }

Partie 3 : Schéma Frontend ↔ Backend (Magic Dictée)

Flux "Magic Dictée" - Dictée Vocale Intelligente

Étape 1 : 🩺 Le Médecin clique sur le bouton "Magic Dictée" et parle dans le micro

↓ Audio enregistré (format Blob)

Étape 2 : 🖥️ Le Frontend envoie l'audio au Backend

POST /api/voice-pilot

↓

Étape 3 : ⚙️ Le Backend (Proxy Sécurisé) traite la requête

→ Appel API 🧠 Groq Whisper : Transcription Audio → Texte

→ Appel API 🧠 Groq Llama 3 : Analyse du texte + Extraction d'actions

↓

Étape 4 : ⚙️ Le Backend retourne au Frontend

{ transcript: "Patient anxieux...", actions: [{ type: "FILL", field: "motif", value: "..." }] }

↓

Étape 5 : 🖥️ Le Frontend exécute les actions

→ Remplit automatiquement les champs du formulaire

→ Affiche la transcription en temps réel

↓

Étape 6 : 🩺 Le Médecin valide et sauvegarde

POST /api/notes → Enregistrement dans 🌩️ Supabase

Partie 4 : Mise en commun rapide (Feedback)

Points forts de notre architecture :

  • Clarté : Chaque rôle a des endpoints dédiés. L'API est RESTful (Noms clairs, Verbes HTTP standards).
  • Cohérence MVP : Pas de fonctionnalités superflues. On se concentre sur l'essentiel : Gérer les patients et dicter les notes.
  • Simplicité : Le backend sert de "passe-plat" intelligent pour sécuriser les clés API de l'IA (Groq) et centraliser les règles de gestion (Rôles).

✔️ Description du Rôle du Backend

  • Orchestrateur Sécurisé : Il vérifie l'identité (Token) avant chaque action.
  • Proxy IA : Il cache la complexité et les clés secrètes des services tiers (Groq, Supabase).
  • Validation Métier : Il applique les règles de gestion (ex: un infirmier ne peut pas modifier un diagnostic).
